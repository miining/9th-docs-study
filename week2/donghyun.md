## 인상 깊었던 핵심 문장

> **"쿠버네티스 오브젝트는 '의도의 기록'이다."**
> 

오브젝트를 생성한다는 것은 곧 쿠버네티스 시스템에 클러스터의 워크로드가 어떤 모습이어야 하는지를 알려주는 것이며, 이것이 클러스터의 **의도한 상태(desired state)** 이다. 선언형(Declarative) 패러다임의 본질을 한 문장으로 정의한 표현이라 인상적이었음.

또한 **권장 레이블(Recommended Labels)** 섹션도 표준화된 레이블 컨벤션이 있다는 점이 흥미로웠다.

## kubectl & kubeconfig

- `kubectl`은 `$HOME/.kube` 디렉터리의 `config` 파일을 찾는다.
- kubectl은 kubeconfig에 정의된 **cluster, user, context** 를 사용해 API 서버에 연결하고 인증한다.
- **클러스터 외부**에서 실행될 때는 kubeconfig에서 API 서버 주소와 인증 정보를 읽음.
- **Pod 내부**에서 실행될 때(예: CI/CD 파이프라인)는 Pod에 마운트된 **ServiceAccount 토큰** 기반의 in-cluster 인증 사용.

## Node 등록 방식

API 서버에 노드를 추가하는 두 가지 방법:

1. **노드의 kubelet이 컨트롤 플레인에 자체 등록(self-registration)**
2. **사용자가 노드 오브젝트를 수동으로 추가**

```json
{
  "kind": "Node",
  "apiVersion": "v1",
  "metadata": {
    "name": "10.240.79.157",
    "labels": {
      "name": "my-first-k8s-node"
    }
  }
}
```

- 쿠버네티스는 유효하지 않은 노드 오브젝트도 일단 **유지**하고, 정상 여부를 계속 확인함.
- 상태 확인을 멈추려면 **명시적으로 노드 오브젝트를 삭제**해야 한다.

## 깊이 파본 개념 #1 — Lease, 그리고 분산 시스템 패러다임

쿠버네티스 문서에서 Lease를 처음 봤을 때는 그냥 "분산 락 같은건가?" 정도로 넘겼는데, 파다 보니 **분산 시스템의 거의 모든 핵심 문제가 여기 다 얽혀 있었다.** 단어 정의로 끝낼 수 없는 주제라서 패러다임 자체를 따라가 봤다.

### 왜 애초에 Lease가 필요한가 — 분산 시스템의 근본 문제

단일 서버 시스템에서는 "지금 누가 락을 잡고 있는지"가 자명하다. 메모리에 변수 하나 두면 끝이다. 하지만 노드가 여러 대로 흩어진 분산 환경에서는 이게 전혀 자명하지 않다.

- **네트워크는 신뢰할 수 없다.** 패킷이 늦게 오거나, 안 오거나, 순서가 뒤바뀐다.
- **노드는 언제든 죽을 수 있다.** 죽은 건지, 느린 건지, 네트워크 단절된 건지 외부에서 구분할 수 없다.
- **시계가 다 다르다.** 노드마다 클럭이 미세하게 어긋난다(Clock Skew).

이 환경에서 "**여러 노드 중 한 명만 어떤 작업을 한다**"를 보장하려면 어떻게 해야 하는가? 이게 분산 시스템의 출발점이고, **Lease는 그 해법 중 하나**다.

### Lease의 핵심 아이디어 — "시간이 만료되는 락"

전통적인 락(mutex)은 누군가 락을 잡고 죽어버리면 영원히 풀리지 않는다(데드락). 분산 환경에서는 노드가 언제든 죽을 수 있기 때문에 **이 방식은 곧바로 시스템을 마비시킨다.**

Lease는 이 문제를 **"락에 만료 시간을 붙이자"** 라는 발상으로 해결한다.

- 누군가 락을 잡되, **TTL(Time-To-Live)** 이 함께 붙는다.
- 락을 계속 들고 있고 싶으면 **주기적으로 갱신(renew)** 해야 한다.
- 갱신 못 하면(=노드가 죽었거나 네트워크가 끊겼으면) **자동으로 락이 풀린다.**
- 다른 노드가 그 락을 새로 acquire 할 수 있다.

즉, **"점유자가 살아있다는 신호(하트비트)와 락을 하나로 묶은 것"** 이 Lease다. 이름이 "임대(Lease)"인 이유도 여기 있다 — 영구 소유가 아니라 **기한부 점유**.

### 쿠버네티스의 Lease 오브젝트

쿠버네티스는 `coordination.k8s.io/v1` API 그룹에 Lease를 1급 리소스로 올려놨다.

```yaml
apiVersion: coordination.k8s.io/v1
kind: Lease
metadata:
  name: my-app-leader
  namespace: default
spec:
  holderIdentity: pod-abc-12345        # 누가 들고 있는지
  leaseDurationSeconds: 15             # 이 시간 안에 갱신해야 유지됨
  acquireTime: "2026-01-30T10:00:00Z"  # 최초 점유 시각
  renewTime: "2026-01-30T10:00:10Z"    # 마지막 갱신 시각
  leaseTransitions: 3                  # 리더가 몇 번 바뀌었는지
```

핵심 필드는 단 세 개다 — **누가(holderIdentity), 언제까지(leaseDurationSeconds), 마지막으로 언제 갱신했는지(renewTime)**. 이게 분산 코디네이션의 최소 단위다.

### 용도 1 — Node Heartbeat

- 모든 Node는 `kube-node-lease` 네임스페이스에 **본인 이름과 매칭되는 Lease 오브젝트**를 하나 가진다.
- kubelet은 주기적으로 이 Lease의 `renewTime` 필드를 업데이트한다. 이게 곧 하트비트.
- 컨트롤 플레인은 timestamp를 보고 노드가 살아있는지 판단.

예전에는 노드가 자기 상태를 알리려고 **Node 오브젝트 전체를 PATCH**했다. Node 오브젝트는 크고 무거우니 etcd에 부하가 컸다. 클러스터가 커질수록 이 부하가 기하급수적으로 증가했다. Lease로 바꾸면서 **작은 오브젝트의 작은 필드만 업데이트** 하면 되니 부하가 극적으로 줄었다. "분산 시스템에서 하트비트는 가능한 한 가벼워야 한다"는 원칙의 좋은 예시다.

### 용도 2 — Leader Election (리더 선출)

여기서부터가 진짜 분산 시스템 패러다임 영역이다.

**왜 리더가 필요한가?** kube-controller-manager나 kube-scheduler를 HA 구성하면 여러 복제본이 뜬다. 그런데 이 복제본들이 **동시에 같은 일을 하면 안 된다.** 예를 들어 두 scheduler가 같은 Pod를 서로 다른 노드에 동시에 할당하려 들면 충돌이 난다. 그래서 "**여러 복제본은 떠 있되, 실제로 일하는 건 한 명**"이라는 패턴이 필요하다 = **Active-Passive HA**.

**Lease 기반 리더 선출 동작 흐름:**

1. 모든 후보가 같은 이름의 Lease를 acquire 시도한다. 가장 먼저 쓴 놈이 리더.
2. 리더는 주기적으로 `renewTime`을 업데이트(= 하트비트).
3. 리더가 `leaseDurationSeconds` 안에 갱신 실패하면 리더십을 잃는다.
4. 다른 후보가 자기 identity를 써넣고 새 리더가 된다.
5. follower들은 backoff를 두고 재시도해서 API 서버에 부하를 주지 않음.

쿠버네티스의 client-go 라이브러리에 `leaderelection` 패키지가 통째로 들어있고, 사용자는 콜백(`OnStartedLeading`, `OnStoppedLeading`)만 구현하면 끝이다. 분산 시스템의 핵심 원시 연산이 **표준 라이브러리 수준에서 추상화**되어 있는 것.

### 그런데 이게 정말 안전한가? — Lease의 함정과 Fencing

여기서 분산 시스템의 가장 악명 높은 함정이 나온다.

**시나리오:**

- 리더 A가 Lease를 들고 있다.
- A가 GC pause(Java 같은 언어에서 흔함)나 네트워크 지연으로 30초간 멈춘다.
- A는 자기가 멈춘 줄 모른다. 깨어나면 자기가 여전히 리더라고 생각한다.
- 그 사이에 Lease가 만료되고 B가 새 리더가 됐다.
- A가 깨어나서 "내가 리더야"라며 작업을 수행한다 → **리더가 둘** = 데이터 손상.

이 문제는 Lease만으로는 완벽하게 해결되지 않는다. 그래서 진짜 강한 보장이 필요한 시스템(데이터베이스 등)은 **Fencing Token**을 같이 쓴다. Lease를 새로 발급할 때마다 **단조 증가하는 토큰 번호**를 함께 발급하고, 작업 대상 시스템(예: 스토리지)은 **더 작은 토큰의 요청을 거부**한다. 깨어난 A의 요청은 옛날 토큰을 들고 있으니 거부된다.

쿠버네티스의 컨트롤러 리더 선출은 **에 가까운 보장**으로 충분하다고 보고 fencing은 따로 안 쓴다. 잘못 동작하더라도 다음 reconcile loop에서 desired state로 수렴하기 때문. 하지만 **etcd 같은 상태 저장 시스템에서는 이 정도 보장으로는 부족**하다. 그래서 더 무거운 합의 알고리즘이 필요해진다.

## 깊이 파본 개념 #2 — Split Brain & Quorum & Raft

Lease는 "한 리더가 살아있는지" 까지는 다루지만, "**네트워크가 진짜로 두 쪽으로 갈라졌을 때 어떻게 할 것인가**"는 다루지 못한다. 이게 분산 시스템의 두 번째 큰 산이다.

### Split Brain 시나리오

![image.png](attachment:09b97ef1-e116-49d2-88e1-d5a66a6ac98e:image.png)

![image.png](attachment:dbc34ddc-763b-405b-831a-60bddc99aea9:image.png)

5노드 클러스터를 멀티 zone에 배치했다고 하자.

- Zone 1: 노드 3개 (N1, N2, N3 — 리더 N1)
- Zone 2: 노드 2개 (N4, N5)

zone 간 네트워크가 끊긴다. 양쪽은 서로 통신할 수 없다.

- Zone 1의 N2, N3 입장: "리더 N1과 통신은 되는데, N4, N5는 안 보이네. 쟤네가 죽었나 보다."
- Zone 2의 N4, N5 입장: "리더 N1이 안 보이네. 죽은 것 같은데. 우리끼리 새 리더 뽑자."

**양쪽이 각자 진행하면** Zone 1은 N1을 계속 리더로 두고 쓰기를 받고, Zone 2는 N4를 새 리더로 뽑아 쓰기를 받는다. 시간이 흘러 네트워크가 복구되면 — **두 그룹이 서로 다른 데이터를 가지고 있다.** 어느 쪽이 진짜인지 판단 불가, 머지도 불가. 이게 split brain이다.

### Quorum — Split Brain의 수학적 해법

![image.png](attachment:ccac4cad-462d-4b68-ab9b-3894dd418605:image.png)

해결책은 단순하면서도 강력하다 — **과반수가 동의해야 결정이 유효하다.**

- 5노드 클러스터의 quorum = 3 (과반수)
- 위 시나리오에서:
    - Zone 1 (3노드): 과반수 확보 → 계속 동작
    - Zone 2 (2노드): 과반수 미달 → **스스로 동작 중단**, 쓰기 거부

핵심은 "**과반수 그룹은 어떤 분할에서도 단 하나만 존재한다**"는 수학적 사실이다. 5명 중 3명이 동의한 그룹과 또 다른 3명이 동의한 그룹은 동시에 존재할 수 없다(비둘기집 원리). 그래서 split brain이 **수학적으로 불가능**해진다.

**왜 홀수 노드가 권장되는가:**

- 짝수(4노드)면 `2 vs 2` 분할 시 양쪽 다 quorum 미달 → 클러스터 전체 마비.
- 홀수(3, 5, 7)면 어떻게 쪼개져도 한쪽은 반드시 과반.
- 사실 4노드는 3노드보다 **장애 허용성이 더 낮다** (둘 다 1대 장애까지만 허용). 그래서 짝수 노드는 비용 낭비.

### CAP 정리와 etcd의 선택

![image.png](attachment:b9765c49-a870-4c24-838b-5e853adfcfa0:image.png)

여기서 **CAP 정리**가 등장한다 — 분산 시스템은 다음 셋 중 최대 두 개만 가질 수 있다.

- **C**onsistency (일관성)
- **A**vailability (가용성)
- **P**artition tolerance (파티션 내성)

네트워크 파티션은 현실에서 반드시 일어나므로 P는 사실상 필수. 그래서 진짜 선택은 **C냐 A냐**다.

- **CP 시스템** (예: etcd, ZooKeeper): 파티션 발생 시 일관성을 지키고, **가용성을 포기** (소수 측은 쓰기 거부).
- **AP 시스템** (예: Cassandra, DynamoDB의 일부 모드): 파티션 발생 시 양쪽 다 계속 받고, **나중에 충돌 해결** (일관성 포기).

**쿠버네티스가 etcd를 선택한 이유**가 여기 있다. 쿠버네티스는 클러스터의 desired state를 저장하는 데이터 스토어가 필요하다. 여기서 데이터가 어긋나면 **두 scheduler가 같은 Pod를 두 노드에 띄우는** 재앙이 일어난다. 그래서 **일관성이 가용성보다 훨씬 중요**하다. etcd는 CP를 선택했다.

### Raft 합의 알고리즘

![image.png](attachment:924aa030-c20a-4ddf-b0b1-201ade14eb00:image.png)

etcd가 quorum을 실제로 어떻게 구현하는가? **Raft 알고리즘**이다. (Paxos보다 이해하기 쉬워서 만들어진 알고리즘.)

Raft의 핵심 아이디어:

1. **리더 기반.** 모든 쓰기는 리더를 통한다. 리더는 quorum 투표로 선출.
2. **로그 복제.** 리더는 쓰기 요청을 받으면 자기 로그에 기록하고, follower들에게 복제 명령을 보낸다.
3. **과반수 응답 후 커밋.** **과반수 follower가 "기록했다"고 응답해야** 리더가 그 로그를 커밋(=클라이언트에 성공 응답).
4. **Term 번호.** 매 리더 임기마다 단조 증가하는 term 번호. 옛 리더가 깨어나서 명령을 보내도 term이 낮으면 거부됨 (= **fencing token과 같은 원리**).

이 원리 때문에 **etcd는 과반수가 살아있는 한 항상 일관된 상태**를 유지한다. 그리고 그 과반수 보장이 곧 쿠버네티스 컨트롤 플레인의 일관성 보장이다.

### Lease와 Raft의 관계

처음에 봤던 Lease로 다시 돌아와 보면, 두 메커니즘의 역할이 명확해진다.

- **Raft (etcd 레벨):** 데이터 자체의 일관성. "어떤 값이 진짜인가"를 합의.
- **Lease (쿠버네티스 컨트롤러 레벨):** 그 위에서 "**누가 작업할 차례인가**"를 조율.

쿠버네티스의 Lease는 etcd 위에 얹혀 있기 때문에, Lease 오브젝트의 읽기/쓰기 자체는 이미 Raft로 일관성이 보장된다. 그래서 Lease는 더 가볍게 "리더 식별"만 담당하면 된다. **계층화된 신뢰**가 잘 드러나는 설계다.

> **AI 답변과 검증의 가치**: "왜 마스터 노드를 홀수로 두냐"는 면접 단골 질문도, 결국 quorum과 split brain을 모르면 외운 답에 그친다. 공식 문서를 훑다 Lease 하나 봤을 뿐인데 분산 시스템 패러다임 전체로 연결됐다.

## 깊이 파본 개념 #3 — cgroup v2의 개선점과 PSI

### cgroup v2의 개선 사항 (v1 대비)

- API의 **단일 통합 계층 구조** 설계 (v1은 cpu, memory, io가 각각 별도 계층)
- 컨테이너에 대한 **더 안전한 서브트리 위임**
- **Pressure Stall Information(PSI)** 같은 새 기능
- 다중 자원에 걸친 **향상된 자원 할당 관리 및 격리**
- 다양한 메모리 할당(네트워크 메모리, 커널 메모리 등)에 대한 **통합 집계**
- **페이지 캐시 라이트백** 같은 비즉각적 자원 변경 집계

### PSI (Pressure Stall Information)란?

Linux 커널 4.20+에서 도입된 기능으로, **CPU/메모리/I/O 자원 부족 때문에 태스크가 stall(멈춰서 기다린) 시간을 측정**해주는 지표. 페이스북이 자기네 데이터센터 운영하면서 만들었음. 기존 load average는 "얼마나 바쁜가"는 보여주지만 "**자원 부족 때문에 멈춘 시간**"은 못 봤다. PSI는 이걸 직접 측정한다.

**측정값은 두 종류:**

| 종류 | 의미 |
| --- | --- |
| **some** | 하나 이상의 태스크가 해당 자원 때문에 stall된 시간 비율 — 자원 부족의 초기 신호 |
| **full** | 모든 non-idle 태스크가 동시에 stall된 시간 비율 — 심각한 자원 부족, 시스템 진행 불가 |

**실제 값 예시 (`/proc/pressure/memory`):**

```
some avg10=0.00 avg60=0.00 avg300=0.00 total=1249184
full avg10=0.00 avg60=0.00 avg300=0.00 total=317955
```

- `avg10` / `avg60` / `avg300` = 최근 10초/60초/300초 동안의 stall 비율(%)
- `total` = 부팅 후 누적 stall 시간(마이크로초)

### cgroup v2 환경에서의 PSI

cgroup v2가 마운트된 환경에서는 **각 cgroup 디렉터리마다** `cpu.pressure`, `memory.pressure`, `io.pressure` 파일이 생긴다. 즉, **컨테이너별로 자원 압박 정도를 따로 측정 가능**하다.

### K8s에서 PSI 활용 — `KubeletPSI`

- **Kubernetes v1.36부터 KubeletPSI 기능 게이트가 `true`로 고정** (안정화 완료).
- kubelet이 PSI 메트릭을 **Node / Pod / Container 레벨**로 수집해서 노출:
    - kubelet **Summary API**
    - `/metrics/cadvisor` 엔드포인트 (Prometheus 포맷)
- 활용 예: 어떤 Pod가 메모리 stall이 잦으면 메모리 부족, I/O stall이 잦으면 디스크 병목. **OOM 발생 전에** 선제 대응 가능.

## 깊이 파본 개념 #4 — MemoryQoS (cgroup v2 메모리 제어의 핵심)

cgroup v2 섹션에 "**일부 쿠버네티스 기능은 향상된 자원 관리 및 격리를 위해 오직 cgroup v2만을 사용한다. 예를 들어, MemoryQoS 기능은 메모리 QoS를 향상시키고 cgroup v2 기본 요소에 의존한다**"라는 문장이 있었다. 이게 그냥 흘려읽기 어려운 이유가 있다 — **쿠버네티스에서 `requests`와 `limits`가 실제로 어떻게 동작하는지에 대한 통념을 뒤집기 때문이다.**

### 통념 깨기 — `requests`는 사실 "예약"이 아니다

많은 사람들이 다음과 같이 착각한다.

> "Pod에 `requests.memory: 1Gi`를 적으면, **그 1Gi만큼은 무조건 보장**되겠지?"
> 

**사실은 아니다.** 기본 쿠버네티스 동작에서 `requests.memory`는 **오직 스케줄러가 어느 노드에 Pod를 배치할지 결정할 때만 사용되는 힌트**다. 일단 Pod가 노드에 떠 버리면, **커널은 이 Pod가 1Gi를 "요청"했다는 사실을 전혀 모른다.** 노드 메모리가 부족해지면 그 1Gi 안에 있는 페이지도 reclaim 대상이 되거나 OOM kill 당할 수 있다.

cgroup v1 시대에는 **"최소 메모리를 보장한다"는 인터페이스 자체가 커널에 없었기 때문에** 어쩔 수 없었다. CPU는 `cpu.shares`로 비율 기반 보장이 가능했지만, 메모리는 `memory.limit_in_bytes`라는 **상한선만** 있고 하한선이 없었다.

### cgroup v2가 가져온 메모리 인터페이스

cgroup v2의 메모리 컨트롤러는 이걸 근본적으로 바꿨다. 네 가지 핵심 노브가 생겼다.

| cgroup v2 노브 | 의미 |
| --- | --- |
| **`memory.max`** | 하드 리밋. 이 값을 넘으면 OOM killer 발동. (cgroup v1의 `memory.limit_in_bytes`와 동일) |
| **`memory.min`** | **하드 보장.** 이 값 이내의 메모리는 **어떤 상황에서도 reclaim 되지 않는다.** 시스템 전체 OOM 위험에 처해도 지킨다. |
| **`memory.low`** | **소프트 보장.** 평소엔 reclaim 안 하지만, **시스템 전체 OOM 위험**이면 reclaim 할 수 있다. |
| **`memory.high`** | **쓰로틀링 임계점.** 이 값을 넘으면 OOM kill 대신 **프로세스를 느려지게 만들고** 강한 reclaim 압박을 가한다. 일종의 "경고 영역". |

핵심 통찰: cgroup v1에서는 메모리가 한계를 넘으면 **곧바로 OOM kill**이었다. v2의 `memory.high`는 **그 사이에 "쓰로틀링"이라는 단계**를 끼워 넣었다. 갑작스러운 메모리 폭증을 부드럽게 흡수할 여지가 생겼다.

### MemoryQoS가 하는 일

**MemoryQoS 기능 게이트(`MemoryQoS: true`)** 를 켜면, kubelet이 `requests.memory`와 `limits.memory` 값을 위 cgroup v2 노브들에 **실제로 매핑해서 커널에게 알려준다.** 통념을 진짜로 만들어주는 기능.

기본 매핑은 다음과 같다.

- `limits.memory` → **`memory.max`** (이건 원래도 됐음)
- `requests.memory` → **`memory.min`** (= **진짜 메모리 보장**)
- `memory.high` = `requests.memory + (limits.memory - requests.memory) × memoryThrottlingFactor`
    - `memoryThrottlingFactor` 기본값은 0.9. 즉 limit의 90% 지점부터 쓰로틀링.

이렇게 되면 비로소 `requests`가 **진짜 예약**이 된다. 커널은 "이 컨테이너의 N바이트는 절대 건드리지 마"라는 명령을 받는다.

### v1.36의 변화 — Tiered Reservation

초기 MemoryQoS는 모든 컨테이너의 request를 `memory.min`으로 매핑했는데, 이게 문제를 일으켰다.

**문제 시나리오:** 8GB RAM 노드에 Burstable Pod들이 총 7GB를 request했다고 하자. 이전 버전에서는 그 7GB가 **전부 `memory.min`으로 잠긴다.** 그러면 커널, 시스템 데몬, BestEffort Pod가 쓸 메모리가 1GB밖에 안 남는다. 오히려 OOM 위험이 더 커진다.

**v1.36의 해법 — QoS 클래스별 차등 보호:**

| Pod QoS 클래스 | 매핑되는 cgroup 노브 | 보호 강도 |
| --- | --- | --- |
| **Guaranteed** | `memory.min` | 하드 보장 (절대 reclaim 안 됨) |
| **Burstable** | `memory.low` | 소프트 보장 (시스템 전체 위험 시에만 reclaim) |
| **BestEffort** | (없음) | 보호 없음, 항상 reclaim 가능 |

이건 쿠버네티스의 **QoS 클래스 철학과 cgroup v2 인터페이스가 깔끔하게 매핑**된 케이스다. "Guaranteed니까 진짜 보장해야지", "BestEffort니까 가장 먼저 희생되어야지"라는 의도가 커널 레벨에서 실제로 강제된다.

### 부모-자식 cgroup 제약과 위임

cgroup v2에는 중요한 규칙이 있다 — **부모 cgroup의 메모리 보호는 자식들의 합 이상이어야 한다.** kubelet은 이걸 유지하기 위해:

- `kubepods` 루트 cgroup의 `memory.min`을 **모든 Guaranteed + Burstable Pod request의 합**으로 설정
- Burstable QoS cgroup의 `memory.low`를 **모든 Burstable Pod request의 합**으로 설정

이렇게 해야 커널이 컨테이너 단위, Pod 단위, QoS 단위에서 일관되게 보호 정책을 강제할 수 있다.

### 알아두면 좋은 함정 — Kernel Livelock

`memory.high` 쓰로틀링에는 알려진 함정이 있다. **커널 5.9 미만 버전**에서는 쓰로틀링이 livelock(시스템이 살아있긴 한데 진행이 안 되는 상태)을 유발할 수 있는 버그가 있었다. 5.9에서 수정됐다. v1.36의 kubelet은 시작 시 커널 버전을 체크해서 5.9 미만이면 경고 로그를 남긴다.

### OOM Group Killing (cgroup v2의 또 다른 변화)

v2에서는 OOM kill 동작도 바뀌었다. cgroup v1은 **컨테이너 안에서 oom_score가 가장 높은 단일 프로세스**만 죽였다. cgroup v2 + Kubernetes 1.28+에서는 `memory.oom.group=1` 설정으로 **컨테이너의 전체 프로세스 그룹을 한 번에** 죽인다.

이게 왜 중요하냐면, **PostgreSQL이나 사이드카처럼 한 컨테이너에 여러 프로세스가 있는 경우** 예전에는 자식 프로세스 하나만 죽고 부모는 살아남아서 좀비 상태가 되는 일이 있었다. 이제는 전체가 깔끔하게 죽고 재시작된다.

### 정리 — MemoryQoS의 의의

핵심 한 줄: **MemoryQoS는 쿠버네티스의 `requests`/`limits` 추상화와 리눅스 커널의 cgroup v2 메모리 컨트롤러를 진짜로 연결시키는 다리**다. 이전에는 사용자가 적은 `requests.memory`가 "스케줄러를 위한 힌트"에 불과했지만, MemoryQoS가 켜진 환경에서는 그게 **커널이 강제하는 진짜 약속**이 된다.

> 공식 문서의 한 줄 ("MemoryQoS는 cgroup v2 기본 요소에 의존한다")이 이렇게까지 깊은 함의를 가지고 있을 줄 몰랐다. 매뉴얼 한 줄을 풀면 그 안에 쿠버네티스의 자원 관리 철학과 리눅스 커널의 진화가 다 들어있다.

## 깊이 파본 개념 #5 — Page Cache Writeback

### Page Cache 기본

- 리눅스 커널은 디스크 I/O를 줄이려고 **물리 메모리의 페이지(보통 4KB) 단위로 디스크 데이터를 캐싱**한다.
- 읽기: cache hit이면 디스크 안 감 / cache miss면 디스크에서 읽어와 캐시에 저장.
- 쓰기: **Write-back 전략** 사용 — 일단 캐시(메모리)에만 쓰고, 디스크에는 나중에 비동기로 반영.

### Dirty Page와 Writeback

쓰기로 수정되었지만 **아직 디스크에 반영 안 된 페이지** = **dirty page** (`PG_dirty` 플래그). 이걸 디스크로 반영하는 과정이 **writeback**.

**Writeback이 일어나는 시점:**

1. **메모리 부족** — 빈 메모리가 임계치 아래로 떨어지면 dirty page를 디스크로 밀어내고 캐시를 비움.
2. **시간 초과** — dirty page가 일정 시간(`vm.dirty_expire_centisecs`, 기본 30초) 이상 오래되면 강제로 기록.
3. **명시적 호출** — `sync()`, `fsync()` 시스템 콜.

**핵심 튜닝 파라미터:**

| 파라미터 | 기본값 | 의미 |
| --- | --- | --- |
| `vm.dirty_background_ratio` | 10% | 전체 RAM 중 이 비율을 dirty가 넘기면 **백그라운드 플러시 시작** |
| `vm.dirty_ratio` | 20% | 이 비율을 넘으면 **새 쓰기가 블록**됨 (강제 동기 플러시) |
| `vm.dirty_expire_centisecs` | 3000 (30초) | dirty page 최대 보유 시간 |

### 왜 K8s 문서에 등장했나

cgroup v2 설명에 "**페이지 캐시 라이트백 같은 즉각적이지 않은 자원 변경에 대한 집계**"라는 표현이 있었다. 이건 다음을 의미한다.

- cgroup v1은 writeback 같은 **비동기 동작**의 메모리 사용량을 cgroup별로 제대로 추적 못 했다. 컨테이너 A가 쓰기를 잔뜩 해서 dirty page가 쌓여도, 실제 flush 시점에는 그게 누구 책임인지 모호해진다.
- cgroup v2는 컨테이너가 만든 dirty page가 결국 누구 책임인지 정확히 추적하고 과금(charge)할 수 있다.
- 결과적으로 컨테이너가 무지막지하게 디스크 쓰기를 해도 **다른 컨테이너에게 그 비용이 전가되지 않는다.** 격리성이 향상됨.
