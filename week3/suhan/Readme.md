#### 서비스 - 엔드포인트 슬라이스

> 엔드포인트슬라이스 API는 서비스가 대규모 백엔드를 처리할 수 있도록 확장할 수 있게 해주고, 클러스터가 정상적인 백엔드의 리스트를 효율적으로 업데이트 할 수 있도록 쿠버네티스가 사용하는 메커니즘

우선 엔드포인트 슬라이스를 이해하기 위해서는 엔드포인트에 대해서 알아야 한다. 엔드포인트는 "**서비스에 연결된 모든 파드의 IP와 포트 정보를 가진 객체**" 이다.
그럼 다시 이름을 확인해보자면. 엔드포인트 슬라이스는 파드가 많아짐에 따라 엔드포인트를 여러개로 슬라이스 해서 저장한다고 볼 수 있겠다.

개발자가 서비스에 셀렉터만 잘 지정해준다면 컨트롤 플레인의 엔드포인트슬라이스 컨트롤러가 클러스터 내의 파드들을 셀렉터 조건과 라벨이 일치하는 것으로 골라내어 오브젝트를 생성하고 기록한다.
Ready와 Serving의 차이

- Ready는 파드가 완전히 기동되어서 신규 트래픽을 정상적으로 받을 수 있는가?
- Serving은 파드가 현재 트래픽을 실제로 처리하고 있는 상태인가?

#### 게이트웨이 API
현재 그리고 앞으로의 쿠버네티스에서 ingress 를 대체한다.

(https://gateway-api.sigs.k8s.io/docs/introduction/)

**게이트웨이 클래스** : 공통 구성을 가진 게이트웨이들의 집합을 정의하며, 해당 클래스를 구현하는 컨트롤러에 의해 관리된다.

**게이트웨이(Gateway)** : 클라우드 로드 밸런서와 같은 트래픽 처리 인프라의 인스턴스를 정의한다

**HTTP라우트(HTTPRoute)** : 게이트웨이 리스너에서 오는 트래픽을 백엔드 네트워크 엔드포인트들의 표현으로 매핑하기 위한 HTTP 전용 규칙을 정의한다. 이러한 엔드포인트들은 종종 서비스로 표현된다.

주요한 부분은 기존의 ingress의 앞부분과 뒷부분을 각각 게이트웨이 / HTTP 라우트로 쪼갠 것으로 보여진다. 가중치 기반 라우팅 / 해더 매칭 /URL 리라이트 같은 기능을 기존 ingress 에서 구현하기위해서는 어노테이션을 달았어야 했는데 이를 공식적인 문법으로 정의하고 역할을 실제 직무위주로 분리한 개념으로 보여진다.

#### 토폴로지 인식 라우팅 / 인식 힌트
Docs 정의
> 토폴로지 인식 라우팅은 네트워크 트래픽이 시작된 영역 내에 유지되도록 돕는 메커니즘을 제공합니다. 클러스터 내 Pod 간에 동일 영역 트래픽을 우선시하면 안정성, 성능(네트워크 지연 시간 및 처리량) 또는 비용 측면에서 도움이 될 수 있습니다.

### 쿠버네티스 스토리지란 뭘까
Docker Volume과의 가장 큰 차이라면 역시 Deployment 파드는 일정하지 않은 노드에 스케쥴된다는 특성이 가장 큰 중점이 아닐까 생각이든다. 또한 그렇기 때문에 언제든 죽고 다시 태어날수 있는 컨테이너의 데이터에 대해서 정합성과 일관성 문제를 해결하기 위해서 존재하는 것으로 생각된다. 즉 **생명주기**가 쿠버네티스 스토리지를 이해한 것에 있어서 매우 중요한 요소라고 생각된다. 스토리지의 생명주기를 파드와 맞출것이냐 아니면 파드 없이도 계속 유지될 것이냐 그리고 워크로드의 특성에 맞게 올바른 스토리지 타입을 최적화 구축할 수 있어야 하는 것으로보인다.


#### 볼륨 - 파이버 채널
파이버 채널 (이하 fc)

> 파이버 채널(Fibre Channel, 줄여서 FC)은 주로 스토리지 네트워킹에 쓰이는 기가비트 속도의 네트워크 기술

쉽게 말해서 스토리지 전용 고속 네트워크 프로토콜이다.
SAN(Storage Area Network) 에 쓰이는 통신망으로,
이걸 쿠버네티스에서 볼륨으로 지정한다는 것은 온프레미스라는 이야기일 것으로 보인다. 쿠버네티스에서의 fc 볼륨이란 스토리지 장비의 디스크 조각인 LUN이 쿠버네티스 파드로 직접꽂아서 pv 로 사용해줄 수 있게 하는 걸 말한다.

WWN(targetWWN) :
World Wide Name으로 파이버 채널 네트워크에 연결되는 모든 하드웨어 장치의 제조 단계에 부여되는 하드웨어 고유 식별 번호이다. 즉, 파이버채널상의 MAC 주소로 보면 된다. 표현 방식도 21:00:00:24:ff:aa:bb:cc 처럼 MAC 과 동일하다.
여기서 WWNN 은 노드의 WWN 을, WWPN은 노드 의 개별 포트에 대한 번호를 가리킨다.

zoning(조닝)이란 :

![](https://velog.velcdn.com/images/victor1919/post/e414d5bc-37d1-437d-8681-55ac8e0421a2/image.png)

SAN 스위치에 연결되어 있는 여러 스토리지의 구역을 나누어주는 것이다.
기본적으로 조닝을 통해 분리된 구역의 경우 서로 간섭하지 못하고 같은 구역 내의 서버와 스토리지끼리만 통신이 가능하다.

zone 1의 경우 스토리지 A를 독점하고 있으며 zone2의 경우 스토리지 B를 스위치 2대에 각각 연결된 서버 B,C,D 가 해당 스토리지를 이용할 수 있다.
zone3는 스위치 2번에 연결된 서버 E,F가 이용한다.
LUN :
LUN(논리 단위 번호)은 데이터 구성 및 액세스를 위해 SAN(Storage Area Network) 환경에서 스토리지 파티션을 정의하는 고유 식별자이다.
파티션이 OS에서의 단위라면 LUN은 그보다 위인 하드웨어에서의 단위로 서버가 스토리지로부터 LUN을 할당받게 되면 서버는 그 LUN 만큼을 하나의 장착된 하드디스크로 인식한다.

#### StorageClass - volumeBindingMode
Immediate
pvc가 생성되는 즉시 그조건에 맞는 pv를 찾아서 바로 바인딩한다.
클라우드환경 혹은 대규모 환겨엥서 볼륨이 만약 특정 존에 먼저 예약이 되어있는데 이 볼륨을 사용해야 할 pod가 다른 구역에 스케쥴링 될수도 있는 문제점이 존재한다. 이렇게 되면 파드 생성시 볼륨을 찾지 못하기 때문에 pod가 생성될 수 없다. 이를 해결하기위해 아래의 WaitForFirstConsumer가 존재한다.

WaitForFirstConsumer
pvc가 생성되어도 즉시 예약을 진행하지 않고 잠깐 멈춰있는다.
그 후 해당 볼륨을 사용하게 될 pod가 파드가 실제로 어느 노드에 배치될지 결정되면 그 노드의 위치에 맞춰 pv를 생성하고 예약을 완료하게 된다.


#### CSI 와 in-tree / Out-of--tree
CSI 는 Container Storage Interface의 약자로 컨테이너 오케스트레이션 시스템을 위한 표준 인터페이스를 정의하여 임의의 스토리지 시스템을 컨테이너 워크로드에 추가하는 인터페이스를 이야기한다.

CSI는 대표적인 아웃트리 플러그인인데, 이 둘을 구분하는 핵심적인 기준은
"스토리지 드라이버 소스 코드가 쿠버네티스 핵심 소스 코드(Core Code) 안에 포함되어 있는가?" 이다.
초창기 CSI 도입전 (인트리) 에는 유명 스토리지 드라이버의 코드가 쿠버네티스 리소스안에 직접 포함되어 컴파일 되었다. 이는 만약 스토리지 벤더가 업데이트를 하면 쿠버네티스 전체의 업데이트 시기까지 기다려야 했으며 스토리지 플러그인을 업데이트하기위해 쿠버네티스 핵심 컴포넌트들이 스토리지 플러그인에 직접적으로 영향을 받았다고 한다. 그렇기 때문에 스토리지의 코드를 쿠번네티스 외부로 완전히 빼내었고 이 방식을 구현하기위한 표준화 된 플러그인이 CSI 인것이다.
