### 구성
구성 챕터는 단순히 cofigmap, secret을 다루는 것이 아닌 쿠버네티스의 자원 리소스 타입 종류, 관리 에 대해서 다룬다.

#### 시크릿을 위한 정보보안
쿠버네티스를 쓰면서 configmap과 secret은 유사하게 환경변수 주입이지만 이름에서 부터 알 수 있듯이 보안 레벨에서 차이가 있는데 어떻게 이루어지는 걸까? 에 대한 생각해본적은 있지만 찾아보지는 않은 의문을 docs를 통해 풀어볼 수 있었는데 당연히 데이터 저장이야 configmap은 평문으로 secret은 Base64로 인코딩 된다는 점 역시 설명하고 있었지만 중요한 2개의 추가 사실을 알 수있었다.

**1. etcd 저장시 암호화의 지원 여부**
configmap : etcd 내 평문 저장
secret : 암호화되어서 etcd 내 저장
백업본 탈취 시에도 비밀번호의 보호가 가능
**2. 노드 메모리 상 관리(tmpfs)**
configmap의 경우 일반적인 설정 파일로 취급되고 노드의 로컬 디스크의 buffer나 파일시스템 레이어에서 일반적으로 취급되는 반면, secret의 경우 kubelet 내부에 시크릿 매니저 모듈이 동작하여 파드가 할당되어 실행될 때만 온디멘드 형식으로 데이터가 전송되고 데이터를 수신할 시 메모리 기반의 임시 가상 파일시스템인(tmpfs) 볼륨을 올려 해당 볼륨에서만 시크릿 데이터는 존재하게 되며 파드가 종료되면 마운트가 해제되어 데이터는 소멸되게 된다. 즉, 시크릿 데이터는 etcd와 실행중인 파드의 메모리에서만 존재하고 그 외에는 절대 쓰여지지 않는다는 것이다.


#### pod 및 container 리소스 - huge page

쿠버네티스의 리소스 타입 중에서 huge page 리소스를 지정할 수 있다고 한다.
**Huge Page**
![](https://velog.velcdn.com/images/victor1919/post/8a0e1d46-8058-473b-aad7-96b6525e91ad/image.png)
리눅스의 기본 페이지 크기는 4kb 이다. 여기서 TLB(Translation Lookaside Buffer) 가 나오게 되는데, TLB는 페이지에 접근하기 위해선 2번의 접근이 필요하기 때문이 이에 생기는 시간을 줄이기 위해서 나온 캐시이다. 그런데 물리 메모리의 성능 향상에 따라 용량이 커지게 되면서 페이지가 만개 십만개 커지게 되면 수많은 TLB 엔트리가 필요해져서 데이터 처리가 느려지는 TLB 미스 현상이 심해진다.
그래서 페이지 자체를 훨씬 크게 늘린 휴즈 페이지가 등장하게 되었다.
Huge page 덕분에 메모리 검색 속도가 비약적으로 상승하였고 메모리 관리를 위해 사용하는 별도의 메모리 공간 자체도 절약되고 또한 huge page 영역은 swap의 대상이 아니라고 한다.
```
apiVersion: v1
kind: Pod
metadata:
  name: hugepages-pod
spec:
  containers:
  - name: db-container
    image: mysql:latest
    resources:
      requests:
        memory: 4Gi
        hugepages-2Mi: 2Gi # 2Mi 크기의 휴즈 페이지를 총 2GiB만큼 요청
      limits:
        hugepages-2Mi: 2Gi # 컨피규레이션 오버헤드를 막기 위해 요청과 제한을 같게 설정합니다.
    volumeMounts:
    - mountPath: /hugepages
      name: hugepage-vol
  volumes:
  - name: hugepage-vol
    emptyDir:
      medium: HugePages # 휴즈 페이지를 파일 마운트 형태로 컨테이너에 제공
```
Huge Page를 cpu나 memory 처럼 쿠버네티스에서 리소스로 사용량을 정해서 파드에서 할당받아 사용할 수 있으며, 이는 노드의 커널에서 미리 생성해두어야 한다.
데이터베이스(Oracle, MySQL), 인메모리 캐시(Redis), 혹은 고성능 AI/빅데이터 연산, 5G 네트워크 통신용 파드 등 대용량 자원을 빠르게 사용해야하는 pod에서 사용되며 huge pages는 overcommit이 불가능하기 때문에 request와 limit가 동일해야한다.

#### 확장 리소스

기본적인 CPU, Memory 이외에 GPU, RDMA 등을 쿠버네티스가 표준 리소스처럼 인식하고 추적할 수 있다. 특히 AI 시대에 GPU 리소스는 매우 중요할 것이다.
하지만 반드시 정수형태로 요청할 수 있으며, Overcommit이 불가능하다는 점이 있다.
그 때문에 역시 request와 limits가 동일해야 한다고 한다.

### 보안

쿠버네티스의 보안은 4C로 구성된다.

#### 멀티-테넨시
처음 들어보지만 Docs의 설명을 빌어 이해해보자면 하나의 데이터센터에 여러 기업의 서버가 입주하듯이 하나의 물리적인 쿠버네티스 클러스터를 여러 팀, 프로젝트, 또는 외부 고객(테넌트)이 공유하여 사용하는 환경으로 이해하였다.
이를 구현하는 방법으로 컨트롤 플레인에서의 네임스페이스, 접근제어, 자원 쿼터 등이 있고 데이터 플레인에서는 네트워크 격리, 스토리지 격리, 노드 격리등이 있다.
하지만 경우에 따라 아예 각 테넨트의 전용 클러스터 할당해야 할 수도 있다.
또한 네임스페이스를 나눌때 어떻게 나눠야 할지도 고려 사항이다. 하나의 조직? 하나의 팀? 하나의 테넨트 당 반드시 하나의 네임스페이스를 할당하는 것으로 충분하지 않을 수 있기 때문이다. 따라서 쿠버네티스에서는 계층적 네임스페이스 컨트롤러를 제공한다고 한다.

### 정책
정책 챕터의 경우 네트워크 정책(Network Policy)는 다루지않고 리소스 정책에 대해서 주로 다루고 있다. 여기서 중요한 것은 제한이다. 쿠버네티스는 기본적으로 클러스터에 네임스페이스, 노드 , 존 등 나누어진 구역별로 제한을 두지않으면 무제한으로 리소스를 점유할 수 있다. 이러한 사고를 방지하기 위해서 쿠버네티스에서는 리밋레인지, 리소스 쿼터 등으로 구역별 자원의 사용 천장을 제한하는 것을 권장한다.
#### 리밋레인지
LimitRange는 실수로 자원설정을 누락하거나, 너무 터무니없이 크거나 작은 값을 지정하는 것을 방지하기 위함이다. 만약 해당하는 네임스페이스의 자원 제한을 아예 적지 않을경우 리밋레인지에 정의해둔 기본 값이 적용된다.
만약 리밋레인지에 의해 설정된 limit의 기본값이 해당 네임스페이스에 스케쥴링 되는 컨테이너의 request값보다 작을 경우 파드는 스케쥴링 될 수 없다.
#### 리소스 쿼터
여러 사용자나 팀이 정해진 수의 노드로 클러스트를 공유할때 더 많은 리소스를 점유할 수 있다. 그렇기 때문에 네임스페이스 단위에서 전체의 총량의 제한을 둘 수 있다.
즉
**리밋레인지** : 네임스페이스에서의 개별 리소스의 규격 및 기본값의 관리
**리소스쿼터** : 네임스페이스 내 모든 자원의 총합

이라고 볼 수 있다. 지원하는 리소스는 컴퓨트, 스토리지, 오브젝트 갯수등 쿠버네티스가 관리하는 대부분의 자원을 설정할 수 있다.


### 스케줄링, 선점, 축출
기본적으로 쿠버네티스에서의 **스케쥴링**이란 kubelet이 파드를 실행할 수 있도록 파드를 노드에 할당하는 행위를 말한다.
**선점**은 우선순위가 높은 파드가 노드에 스케쥴링 될 수 있도록 우선순위가 낮은 파드를 종료시키는 과정을 나타내며, **축출**은 리소스가 부족한 노드에서 하나 이상의 파드를 사전에 종료시키는 프로세스를 의미한다.

여기서 쿠버네티스를 처음 공부했을 때가 생각이 나는데, 쿠버네티스가 자동으로 알아서 최적의 노드를 찾아서 리소스를 자동으로 스케쥴링해주는데 왜 튜닝을 해야하고 공부를 따로 해야할까? 라고 생각했었는데 특히 AI 서버가 중요한 최근, 특수 하드웨어 노드에 특정 파드의 배치, 상황에 따라 알맞은 노드의 파드 배치 토폴로지에서의 통신최적화 스케쥴링, 파드의 중요성 등등... 쿠버네티스 스케쥴러는 기본적으로 노드의 남은 공간만을 최적화 시켜 자원을 배치하기 때문에 인프라의 성격과 목적에 따라 최적화를 위해 튜닝이 필요하기 때문에 이를 공부해야한다고 docs를 보며 생각이 되었다.


#### 필터링 & 스코어링
kube- scheduler의 파드에 대한 노드를 선택하는 단계는 2단계를 거친다.

1. 필터링
   파드를 스케쥴링 할 수 있는 노드 모음을 찾는다.
   PodFitsResources 필터 : 후보 노드가 파드의 특정 리소스 요청을 충족 시키기에 충분한 가용 리소스가 남았는지 확인한다.

2. 스코어링
   필터링을 거친 노드 사이에서 노드의 순위를 지정하여 가장 적합한 파드 배치를 찾아낸다. 스케쥴러는 사용 중인 스코어링 규칙에 따라 필터링에서 통과된 각 노드에 대한 점수를 지정한다.

이 두단계를 거치게 되면 kube-scheduler는 순위가 가장 높은 노드에 pod를 할당한다. 만약 NodeName을 통해 명시적으로 노드를 직접 선택하는 경우를 제외하고는 나머지의 간접 지정 케이스에서는 1차적으로 (nodeSelector 나 nodeAffinity 등) 적합한 노드를 선별한 후 해당 과정이 진행된다.


#### 토폴로지 분배 제약 조건

대규모 환경에서의 토폴로지에서 파드의 분배 방식을 제어한다.
**topologySpreadConstraints 필드**
토폴로지 분배 제약조건을 구성하는 필드이며 여기서 중요한 키는 바로 **maxSkew** 이다.
maxSkew(최대 허용 불균형도) : 도메인 간 파드 개수의 최대 차이 수치를 정의.
토폴로지 키(분배 제약을 거는 단위)가 Zone 일때, DoNot스케쥴로 조건을 걸고
만약 maxSkew가 1이라면? Zone A에 파드가 2개 있을 때 Zone B에는 최소 1개나 최대 3개까지만 올 수 있다.

![](https://velog.velcdn.com/images/victor1919/post/71cab33f-2c89-41c6-a463-7e5077849d28/image.png)

위 그림을 보면 알겠지만 ZonA가 파드가 3개가 되는 순간 ZoneB의 파드는 1개 밖에 없음으로 이를 어기기때문에 ZoneB로 스케쥴링 될 수 밖에 없다.

만약 토폴로지 키를 Zone 과 node 로 다중 분배 제약을 건다면?
둘다 만족하는 노드에 배치되게 된다. 만약 상충하게 된다면? 파드는 스케쥴 될 수 없다.

#### 선점 과 축출
선점과 축출 둘 다 결국 기존에 Running이던 pod를 종료시킨다는 점에서 동일하게 보이지만 어떤 차이가 있는 걸까?

선점(Preemption)은 컨트롤 플레인의 kube-scheduler에 의해서 제어된다.
파드가 스케쥴링 될 때 클러스터 전체의 여유 공간이 부족할 시 현존하는 파드 중 새로 스케쥴링될 파드보다 우선수위 수치가 낮은 파드를 종료 시키고 배포해야하는데 이 때 저우선 순위 파드의 개수나 자원규모가 가장 적은 노드를 선택하게 된다.
그 후 선택된 노드의 PriorityClass가 낮은 pod를 Graceful shutdown시킨다.

반면, 축출은 데이터 플레인의 kubelet이 독립적으로 시행한다.
가동중인 노드의 임계값에 자원이 압박이 오게 되면 노드의 생존을 위해서 파드를 종료시키게 되며 이 기준에는 Qos 클래스, 실제 메모리 사용량, request 초과량 등이 고려된다.(파드의 자원 사용량이 요청량을 초과했는지 여부, 파드 우선순위, 파드의 자원 요청량 대비 자원 사용량) 또한 축출 임계 값을 (soft, hard) 기준으로 정해서 하드 축출을 해야하는 상황일 경우 유예기간없이 바로 SIGKILL 신호를 보내 프로세스를 강제로 죽이게된다.
